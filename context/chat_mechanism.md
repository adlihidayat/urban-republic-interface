## 1. What This Project Is

A **sales agent API** deployed on HuggingFace Docker Spaces. The agent handles:

- Working memory
- Internal knowledge base
- Chat history storage (Supabase)
- Tool usage during inference

The frontend (Next.js 16) is a **web chat UI** that lets users interact with the agent.

## 2. API Contract

### Health Check

```
GET /health
```

Use this to check if the HuggingFace space is awake before making real requests. HF free spaces sleep when idle — ping health first and wait for 200 before proceeding.

### Chat

```
POST /api/v1/chat
```

**Request Body:**

```json
{
  "message": "hi!",
  "session_id": "test-001",
  "user_id": "test@example.com"
}
```

**Response:**

```json
{
  "response": "Hi there! Welcome to Urban Republic...",
  "intent": "small_talk",
  "session_id": "test-001"
}
```

> Returns a **single response only**, not the full conversation history.  
> The agent manages history internally on the backend (Supabase).

---

### Chat History

```
GET /api/v1/history/{user_id}
```

> ⚠️ **Important:** History is keyed by `user_id` (email), NOT `session_id`.  
> `session_id` returns as `null` — the backend does not store session per message.  
> You always get the full user history, across all sessions.

**Response shape:**

```json
{
  "user_id": "test@example.com",
  "session_id": null,
  "messages": [
    {
      "id": 25,
      "role": "user",
      "content": "hi!",
      "intent": "",
      "created_at": "2026-03-16T07:52:13.512529+00:00"
    },
    {
      "id": 26,
      "role": "assistant",
      "content": "Hello! It's great to meet you...",
      "intent": "small_talk",
      "created_at": "2026-03-16T07:52:14.000000+00:00"
    }
  ]
}
```

**Frontend mapping note:**  
`role` is already `"user"` or `"assistant"` — no transformation needed.  
Use `created_at` as timestamp. `id` and `intent` can be stored but are not required for display.

## 3. Memory Architecture

The agent uses **two separate memory layers** with different purposes.

### Layer 1 — Working Memory (Redis / Upstash) — HOT

```
session:{session_id}:messages   → last 10 messages (5 turns)
session:{session_id}:state      → intent, locale, cart, user profile
session:{session_id}:summary    → distilled summary of older context
cache:tool:{name}:{hash}        → tool result cache (product=5min, policy=60min)
```

- Keyed by `session_id` (generated by the **frontend**)
- TTL: **60 minutes of inactivity** — deleted from Redis after that
- Gives the agent its short-term conversational context per session
- Without the correct `session_id`, the agent has no memory of prior messages in the session

### Layer 2 — Chat History (Supabase) — COLD

```
table: chat_messages
columns: id, user_id, session_id, role, content, intent, created_at
```

- Written **async (fire-and-forget)** after every turn — never blocks the response
- Keyed by both `user_id` and `session_id` in the DB, but the history API queries by `user_id` only
- Never expires — permanent audit trail
- This is what `GET /api/v1/history/{user_id}` reads from

### Why `session_id` is null in the History API response

The `session_id` **is written correctly** to Supabase on every turn (confirmed from `ChatHistoryService.save_turn()`). The null in the API response is a **bug in the history endpoint** — the serializer or SELECT query is not returning `session_id`. The data exists in the database. This is a backend fix, no frontend impact.

### Who Owns What

| Concern                           | Owner                                   |
| --------------------------------- | --------------------------------------- |
| Short-term context (last 5 turns) | Redis, keyed by `session_id`            |
| Permanent chat audit trail        | Supabase `chat_messages`                |
| Tool result caching               | Redis, short TTL                        |
| Display messages array            | Frontend local state                    |
| `session_id` generation           | **Frontend** — backend never creates it |
| `session_id` persistence          | Frontend `localStorage`                 |

**Critical rule:** The frontend never touches Redis or Supabase directly. All memory is managed exclusively through the agent API.

## 4. Frontend Chat Flow (Full)

```
1. User logs in with email
        ↓
2. App generates session_id (uuid) — stored in localStorage
        ↓
3. App pings GET /health → wait for space to wake up if sleeping
        ↓
4. App fires two parallel requests:
   - GET /api/v1/history/{user_id}   → load chat history (keyed by email)
   - GET /api/v1/products            → load product list (for product section)
        ↓
5. UI renders:
   - If history exists  → show previous conversation
   - If no history      → show generic fresh welcome message
        ↓
6. User types a message
        ↓
7. Optimistic UI: append user message to local messages array immediately
        ↓
8. POST /api/v1/chat  { message, session_id, user_id }
        ↓
9. Agent processes → returns single response
        ↓
10. Append agent response to local messages array
        ↓
11. Loop back to step 6
```

## 5. Frontend State Model

```ts
type Message = {
  role: "user" | "assistant";
  content: string;
  timestamp: string;
};

// Managed with useState or useReducer
const [messages, setMessages] = useState<Message[]>([]);
```

- `session_id` → generated with `uuid`, stored in `localStorage`
- `user_id` → from auth (email on login)
- Messages array is the **single source of truth** for the chat UI display

## 6. History Endpoint

The history endpoint **exists** and is keyed by `user_id` (email), not `session_id`.

```
GET /api/v1/history/{user_id}
```

**Key behaviors:**

- History persists across all sessions for the same user
- A user who logs in on a new device still gets their full history
- `session_id` appears as `null` in the API response — this is a backend serializer bug, not a data issue. The value is stored correctly in Supabase.
- Frontend gets the full user history in one call — no pagination observed yet

**On page load**, frontend calls this endpoint and maps `messages[]` into the display array. If `messages` is empty or the array has no items, show the welcome screen instead.

## 7. Groq API Key — User-Supplied Key Feature

### Goal

Allow users to optionally supply their own Groq API key to bypass the free-tier rate limit on the shared server key.

### Decided Approach: httpOnly Session Cookie

When user submits their key on login, the **Next.js server** stores it in a `httpOnly` cookie — never in React state, never in localStorage, never in the database.

```
User enters key on login form
        ↓
Next.js server action receives it
        ↓
Sets httpOnly session cookie (expires on browser close)
        ↓
On each chat request, Next.js API route reads cookie server-side
        ↓
Forwards key as X-Groq-API-Key header to HuggingFace
        ↓
Key never touches frontend JS or your DB
```

### Why httpOnly Cookie Over Other Options

| Option                    | Security             | UX                           |
| ------------------------- | -------------------- | ---------------------------- |
| React state only (memory) | ✅ Best              | ❌ Lost on refresh           |
| localStorage              | ❌ XSS vulnerable    | ✅ Persists                  |
| Encrypt + localStorage    | ⚠️ Decent            | ⚠️ Need password             |
| httpOnly cookie           | ✅ JS cannot read it | ✅ Persists until tab closes |

### Next.js Acts as Proxy

The frontend **never calls HuggingFace directly**. All requests go through a Next.js API route, which:

1. Reads the Groq key from the httpOnly cookie server-side
2. Forwards the request to HuggingFace with the key in the header
3. Returns the response to the browser

This also hides your HuggingFace URL from the public.

```
Browser → Next.js API route → HuggingFace agent
```

### Backend Side

On the HuggingFace agent, check the incoming header:

```python
groq_key = request.headers.get("X-Groq-API-Key") or DEFAULT_GROQ_KEY
```

### Key field is optional

If the user does not provide a key, the backend falls back to the shared server key. No error.
